GLOBAL STRATEGY NOTE:

This plan is focused on reaching a stable, market-ready accounting SaaS.

Becoming the world’s #1 accounting software will require additional layers after this plan, including:
- advanced financial intelligence
- predictive analytics
- multi-country accounting systems
- high-scale performance optimization
- superior UX and onboarding

Do NOT prematurely optimize for these.
First complete all phases in this plan and reach Public Beta readiness.


TETAVIO MASTER PRODUCT PLAN

Strategic Goal:
Tetavio must become a market-ready, trustworthy accounting SaaS platform first, and then evolve toward becoming the world’s #1 accounting software.

Core Principles:
1. Financial accuracy comes before feature quantity.
2. Backend is always the source of truth.
3. Frontend must never send accountId.
4. Invoice totals and accounting totals must be backend-derived.
5. No production data should ever be reset.
6. Never run prisma migrate reset on production.
7. Every change must preserve auth, payment, invoice calculation, and accounting integrity.
8. Build after every meaningful change.
9. Update this plan.txt after each completed phase.
10. Update AGENTS.md only when architecture, API contracts, security rules, or project structure changes.

Current Product Status:
- MVP readiness: approximately 70–75%
- Stable public SaaS readiness: approximately 55–60%
- World-class accounting platform readiness: early foundation stage

Already Completed:

Important Deployment Rules:
- Frontend: run npm run build, then upload contents of dist/ to Hostinger public_html.
- Do not upload the dist folder itself.
- Backend: Render auto-deploys from GitHub.
- Production backend build command must include:
  npm install --include=dev && npx prisma generate && npx prisma migrate deploy && npm run prisma:seed && npm run build
- Production database data should not be deleted by normal build/deploy.
- Never use prisma migrate reset on production.
- Local dev database reset is allowed only with explicit confirmation.

RULES FOR CLAUDE / CODEX

Before starting any task:
1. Read AGENTS.md.
2. Read plan.txt.
3. Follow the next incomplete phase unless explicitly instructed otherwise.
4. Do not rewrite architecture without permission.
5. Keep changes incremental.

During work:
1. Do not modify auth/session/token logic unless task explicitly requires it.
2. Do not modify payment/subscription logic unless task explicitly requires it.
3. Do not modify invoice calculation logic unless task explicitly requires it.
4. Do not send accountId from frontend.
5. Do not reintroduce localStorage for ERP/accounting data.
6. Do not expose passwordHash, tokens, payment payloads, or secrets.
7. Use explicit Prisma select where sensitive data may exist.
8. Keep backend as source of truth.

After completing a phase:
1. Run backend build.
2. Run frontend build.
3. Update AGENTS.md only if architecture/API/security/project rules changed.
4. Update plan.txt:
   - mark phase as COMPLETE
   - add commit hash
   - add short note of what changed
   - update next priority if needed
5. Commit only after builds pass.
6. Push to origin/main.
7. Report:
   - files changed
   - endpoints added/updated
   - migration name if any
   - build results
   - commit hash
   - what remains incomplete

Production safety:
1. Never run prisma migrate reset on production.
2. Production deploy uses prisma migrate deploy only.
3. Render/Hostinger deploy should not delete user data.
4. Any destructive DB command requires explicit confirmation and must be local/dev only.

Rules:
